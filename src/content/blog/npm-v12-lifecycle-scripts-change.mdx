## 一个被忽视了十几年的安全隐患

很多人以为 `npm install` 只是把代码下载到 `node_modules`。但实际上，在安装过程中 npm 还会自动执行依赖包声明的**生命周期脚本**。

例如：

```json
{
  "scripts": {
    "postinstall": "node install.js"
  }
}
```

安装完成后，npm 会自动执行这段代码。

理论上来说，这些脚本拥有与当前用户相同的权限，可以读取 SSH Key、GitHub Token、环境变量，访问文件系统，甚至直接执行系统命令。

换句话说，当你运行 `npm install` 时，实际上是在**执行来自互联网的第三方代码**。

而更离谱的是，这个行为过去十几年一直**默认开启**。

## 为什么 GitHub 现在才动刀？

答案很简单：**供应链攻击已经越来越严重了。**

过去几年，几乎每隔几个月就会爆出一次 npm 投毒事件。从 `event-stream`、`ua-parser-js`、`coa`、`rc`，到最近的 Nx Console 投毒事件、GitHub 内部仓库泄露事件，攻击者越来越喜欢利用生命周期脚本作为传播入口。

因为对于攻击者来说，最理想的攻击方式并不是等待用户运行代码，而是在**安装阶段**就完成感染。

攻击路径通常是这样的：

```
发布恶意包
  ↓
开发者执行 npm install
  ↓
postinstall 自动执行
  ↓
窃取 Token 或凭证
  ↓
控制维护者账号
  ↓
发布更多恶意包
  ↓
继续传播
```

最可怕的是，很多被利用的依赖甚至从来不会被真正运行。它可能只是一个 TypeScript 类型定义包、一个构建工具插件，甚至只是开发依赖，但只要进入安装流程，恶意脚本就有机会执行。

这意味着**感染发生在安装阶段，而不是运行阶段**。

## npm v12 真正解决了什么？

很多人看到这个更新后，会认为 npm 变得更安全了。这种说法没错，但并不准确。

更准确地说，npm v12 真正做的事情是：**降低整个 JavaScript 生态的传播能力。**

React 社区著名技术博主 Sebastien Lorber（ThisWeekInReact）提出了一个非常有意思的观点。他认为 npm v12 更像是一针**疫苗**——它并不能彻底消灭恶意包，但可以降低恶意包传播的效率。

疫情期间大家都听过一个概念：**R 因子**，也就是传播系数。

当 `R > 1` 时，疫情会持续扩散；当 `R < 1` 时，疫情会逐渐消失。

供应链攻击其实也是同样的逻辑。过去的传播链是这样的：

```
恶意包 → 感染开发者 → 窃取维护者账号 → 发布更多恶意包 → 感染更多开发者
```

而 npm v12 的作用，就是直接切断最容易传播的一环：**自动执行安装脚本**。

当感染开发者的成本变高之后，维护者账号被盗的概率会下降，恶意包的数量也会下降，最终整个生态都会变得更安全。

## 不用 npm 的人为什么也应该庆祝？

很多开发者会觉得：我用 pnpm，我用 Bun，我用 Yarn，那 npm 的更新和我有什么关系？

实际上关系非常大。

因为 npm 保护的并不仅仅是 npm 用户，而是整个 JavaScript 开源生态。想象一下，一个维护者的电脑因为恶意 postinstall 被攻破：

```
GitHub Token 被盗
  ↓
发布恶意版本
  ↓
同步到 npm Registry
  ↓
被 pnpm 安装
  ↓
被 Yarn 安装
  ↓
被 Bun 安装
```

最终影响的是**所有开发者**。

Sebastien 提到的一个观点非常值得思考：**即使你不用 npm，你也应该庆祝 npm 的安全升级。** 因为大量开源维护者依然在使用 npm，保护他们，其实也是在保护整个生态。

## 哪些项目会受到影响？

这次变更虽然意义重大，但也会带来不少兼容性问题。尤其是依赖安装脚本完成初始化的项目。

例如：`sharp`、`canvas`、`sqlite3`、`bcrypt`、`better-sqlite3`、`node-sass` 这些原生模块通常需要通过 `node-gyp rebuild` 完成本地编译。

除此之外，很多现代工具链也依赖 postinstall：

- Prisma Client 生成
- OpenAPI SDK 生成
- Codegen 工具
- Rust Binding 下载
- Native Binary 安装
- Monorepo 初始化流程

未来这些场景都需要开发者**主动授权执行**。GitHub 已经在 npm 11.16 中加入兼容性提示，方便开发者提前发现问题并完成迁移。

## npm 只是开始

如果把时间线拉长来看，会发现这并不是 npm 一家的决定。实际上，整个 JavaScript 生态都在朝着同一个方向演进。

目前已经开始默认限制安装脚本的工具包括：

- **npm v12**
- **pnpm 10**
- **Yarn 4**
- **Bun**
- **Deno**

大家正在逐渐形成一个新的共识：**默认拒绝 → 按需授权 → 最小权限原则**。

这个思路其实和现代浏览器越来越像。浏览器不会默认允许网页读取你的定位、访问你的摄像头或者麦克风，而是需要用户主动授权。JavaScript 生态也正在向同样的安全模型靠拢。

---

## Node.js 的下一步会是什么？

Sebastien 在文章中还提到了一个值得关注的方向：**Node.js 权限模型**。

今天的 Node 默认拥有完整权限。任何依赖理论上都可以：

- 访问文件系统
- 发起网络请求
- 读取环境变量
- 创建子进程
- 执行系统命令

但未来很可能会出现类似 Deno 的权限机制：

```bash
node app.js --allow-net
node app.js --allow-read
```

甚至：

```bash
node app.js --deny-fs
```

让应用只能获得必要权限。如果这一步最终落地，那么 Node.js 的安全模型将迎来比 npm v12 更大的变化。

---

## 写在最后

过去十几年里，整个生态默认相信第三方代码。开发者执行一次 `npm install`，就意味着允许陌生代码在自己的电脑上运行。

而今天，npm、pnpm、Yarn、Bun、Deno 正在共同推动另一套规则：

> **安装依赖 ≠ 执行代码**
> **默认拒绝，按需授权**

npm v12 虽然无法彻底消灭供应链攻击，但它很可能会成为 Node.js 生态安全史上的一个标志性节点。
