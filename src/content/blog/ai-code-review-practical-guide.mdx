## Background

The open source projects I maintain receive an average of 10-15 PRs per day. Manually reviewing them one by one would eat up my entire day.

Six months ago, I started experimenting with AI as a code review assistant, and I've now built a fairly mature workflow.

This post shares my hands-on experience—**what AI does well, and what humans must handle**.

## What AI Code Review Is Good At

### 1. Style and Convention Checks

AI is incredibly sensitive to code conventions, and it **never gets tired**:

```typescript
// ❌ Issues AI will catch
function fetchData(url){  // missing space
  return fetch(url).then(res=>res.json())  // inconsistent arrow function style
}

// ✅ AI suggestion
function fetchData(url: string) {
  return fetch(url).then((res) => res.json());
}
```

**Best practice**: Feed AI your project's ESLint/Prettier config, naming conventions, and commit conventions. Let it be the first line of defense.

### 2. Common Bug Patterns

AI is great at spotting known bug patterns:

- Unhandled async errors
- Memory leaks (uncleaned listeners, timers)
- Type assertion abuse (`as any`, `as unknown as X`)
- Duplicate code (DRY principle violations)
- Unsafe user input handling

### 3. Test Coverage Analysis

```typescript
// AI review findings:
// ⚠️ New errorHandler function added, but no corresponding test cases
// ⚠️ getUser return format changed, but existing tests don't cover new fields
// 💡 Suggestion: Add errorHandler.test.ts, update getUser mock data
```

### 4. Docs-vs-Code Consistency

AI can spot "the comments say one thing, the code does another"—something human reviewers最容易 overlook.

## Limitations of AI Code Review

### ❌ Architectural Decisions

AI can't judge whether "this module should go in this directory" or "should we use composition or inheritance." It lacks understanding of the project's overall evolution.

### ❌ Business Logic Correctness

AI doesn't know your business rules. It might think the code looks beautiful, but the logic could be completely wrong.

### ❌ Team Politics and Code Ownership

"Which team should maintain this module?" "Will this change affect other teams?"—these require human judgment.

### ❌ Aesthetics and Taste

Code conventions and style can be automated, but **code taste**—knowing when to abstract vs. when to be explicit—AI is nowhere close to getting that right.

## My AI Code Review Workflow

### Step 1: AI First Pass (Automated)

When a PR is submitted, AI automatically does the first round of review:

```
AI Review Checklist:
✅ Code style follows project conventions
✅ No obvious security vulnerabilities
✅ New code has corresponding tests
✅ Public APIs have JSDoc comments
✅ No unnecessary dependencies introduced
⚠️ Found 3 potential performance issues (see below)
⚠️ Suggestion: handleSubmit function is too long (85 lines), consider splitting
```

### Step 2: Deep Human Review

After AI flags the issues, I **only review the flagged sections + critical logic**, instead of reading through the entire diff.

This saves me roughly 60% of review time.

### Step 3: AI Re-Review

After I finish reviewing, I have AI check again:

```
"Compare the original PR with my review comments,
check if I missed any important issues,
and whether my review comments are reasonable."
```

This step often catches things I missed, or makes me realize some of my review comments were overly strict.

## Practices for Open Source Projects

For open source projects, I've set up an AI review bot with these rules:

```
## Auto-approved PRs (AI approves directly)
- Documentation/comment changes only
- Test case changes only
- Dependency version updates (patch level)

## PRs requiring human review
- New features/APIs
- Core logic changes
- Database schema changes
- Build configuration changes

## Auto-rejected PRs
- Contains sensitive information (keys, tokens)
- Binary file changes
- Major lock file changes
- Code changes without tests
```

## Wrapping Up

AI Code Review isn't about replacing human reviewers—it's about **letting human reviewers spend their energy where it matters most**.

- ✅ AI handles style checks, common bug detection, test coverage analysis
- ❌ Leave architectural decisions, business logic, and code taste to humans
- 🔄 Form a closed loop: "AI first pass → human deep review → AI re-review"

**Remember**: AI review helps you **discover** issues, not **judge** them. The final responsibility is still yours.
