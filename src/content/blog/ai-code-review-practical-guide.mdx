## 背景

我维护的几个开源项目，平均每天收到 10-15 个 PR。靠人工一个个 review，一天的时间就搭进去了。

半年前我开始尝试用 AI 做代码审查的辅助，到现在已经形成了一套相对成熟的工作流。

这篇文章分享我的实战经验——**哪些 AI 能做好，哪些必须人来做**。

## AI Code Review 擅长什么

### 1. 规范性检查

AI 对代码规范极其敏感，而且**不会累**：

```typescript
// ❌ AI 会发现的问题
function fetchData(url){  // 缺少空格
  return fetch(url).then(res=>res.json())  // 箭头函数不规范
}

// ✅ AI 建议
function fetchData(url: string) {
  return fetch(url).then((res) => res.json());
}
```

**最佳实践**：把项目的 ESLint/Prettier 配置、命名规范、commit 规范都告诉 AI，让它做第一道把关。

### 2. 常见 Bug 模式

AI 对已知的 bug 模式识别能力很强：

- 未处理的异步错误
- 内存泄漏（未清理的监听器、定时器）
- 类型断言滥用（`as any`、`as unknown as X`）
- 重复代码（DRY 原则违反）
- 不安全的用户输入处理

### 3. 测试覆盖率分析

```typescript
// AI review 时发现：
// ⚠️ 新增了 errorHandler 函数，但没有对应的测试用例
// ⚠️ 修改了 getUser 的返回格式，但现有测试没有覆盖新字段
// 💡 建议：补充 errorHandler.test.ts，更新 getUser 的 mock 数据
```

### 4. 文档与代码的一致性

AI 能发现"注释说的和代码做的不一样"的问题——这是人类 review 时最容易忽略的。

## AI Code Review 的局限性

### ❌ 架构决策

AI 很难判断"这个模块应该放在哪个目录"、"应该用组合还是继承"这类架构问题。因为它缺乏对项目整体演变的了解。

### ❌ 业务逻辑正确性

AI 不知道你的业务规则。它可能觉得代码写得很漂亮，但逻辑完全是错的。

### ❌ 团队政治和代码所有权

"这个模块应该由哪个团队维护"、"这个改动会不会影响其他团队"——这些需要人的判断。

### ❌ 审美和品味

代码规范和风格可以自动化，但**代码的品味**——什么时候该抽象、什么时候该直白——AI 还远做不到。

## 我的 AI Code Review 工作流

### 第一步：AI 初筛（自动）

当 PR 提交时，AI 自动做第一轮 review：

```
AI Review Checklist:
✅ 代码风格符合项目规范
✅ 没有明显的安全漏洞
✅ 新增代码有对应的测试
✅ 公共 API 有 JSDoc 注释
✅ 没有引入不必要的依赖
⚠️ 发现 3 处潜在的性能问题（见下方）
⚠️ 建议：函数 handleSubmit 过长（85行），建议拆分
```

### 第二步：人做深度 review

AI 标记出问题后，我**只 review AI 标记的部分 + 关键逻辑**，而不是通读整个 diff。

这大概节省了我 60% 的 review 时间。

### 第三步：AI 复查

我 review 完后，让 AI 再检查一遍：

```
"请对比原始 PR 和我的 review 意见，
检查我有没有遗漏重要问题，
以及我的 review 意见是否合理。"
```

这一步经常能发现我漏掉的问题，或者让我意识到某些 review 意见过于苛刻。

## 开源项目中的实践

对于开源项目，我设置了一个 AI review bot，规则如下：

```
## 自动通过的 PR（AI 直接 approve）
- 只修改了文档/注释
- 只修改了测试用例
- 依赖版本更新（patch 级别）

## 需要人工 review 的 PR
- 新增功能/API
- 修改了核心逻辑
- 修改了数据库 schema
- 修改了构建配置

## 自动打回的 PR
- 包含敏感信息（密钥、token）
- 二进制文件变更
- 大幅修改了 lock 文件
- 没有测试的代码变更
```

## 总结

AI Code Review 不是要取代人类 reviewer，而是**让人类 reviewer 把精力花在最有价值的地方**。

- ✅ AI 做规范性检查、常见 bug 检测、测试覆盖分析
- ❌ 架构决策、业务逻辑、代码品味留给人类
- 🔄 形成"AI 初筛 → 人深度 review → AI 复查"的闭环

**记住**：AI review 是帮你**发现**问题，不是帮你**判断**问题。最终的责任还是在你手上。
