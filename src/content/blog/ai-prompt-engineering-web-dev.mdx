## 一、Web 开发 Prompt 的核心框架

我总结了一个 **4C 框架**，适用 90% 的 Web 开发场景：

| 要素 | 含义 | 示例 |
|------|------|------|
| **Context** | 项目上下文 | 技术栈、已有代码、约束条件 |
| **Command** | 明确指令 | 要做什么、输出格式 |
| **Constraint** | 约束条件 | 性能要求、兼容性、安全要求 |
| **Check** | 检查要求 | 让 AI 自我检查、指出风险 |

### 实战对比

```
❌ 差 Prompt：
"帮我写一个登录页面"

✅ 好 Prompt：
Context: Next.js 16 + TypeScript + Tailwind 项目
Command: 创建一个登录页面组件，包含邮箱、密码输入框和提交按钮
Constraint: 
- 表单验证（邮箱格式、密码长度 >= 8）
- 加载状态
- 错误提示
- 响应式设计（移动端适配）
Check: 生成后请指出这个实现可能存在的安全问题
```

## 三、高级 Prompt 技巧

### 3.1 Chain of Thought（思维链）

让 AI 展示推理过程，而不是直接给答案：

```
"请一步步分析这个 bug 的可能原因，
先列出所有可能性，
然后逐一排除，
最后给出最可能的根因和修复方案。"
```

**效果：** AI 的准确率提升 30-50%。

### 3.2 Few-Shot（少样本学习）

给 AI 一个你想要的输出示例：

```
"请按照以下格式输出代码审查意见：

## 问题类型：[性能/安全/可维护性]
## 问题描述：[简短描述]
## 代码位置：[文件名:行号]
## 建议修复：[具体建议]
## 优先级：[高/中/低]

示例：
## 问题类型：性能
## 问题描述：useEffect 缺少依赖项导致无限循环
## 代码位置：src/hooks/useData.ts:15
## 建议修复：将 fetchData 加入依赖数组
## 优先级：高

现在请 review 以下代码..."
```

### 3.3 Negative Prompt（反向约束）

告诉 AI **不要**做什么：

```
"不要使用 any 类型
不要引入新的依赖
不要修改现有的 API 签名
不要生成不必要的注释"
```

### 3.4 角色扮演

```
"你是一个有 10 年经验的 Next.js 开发者，
你参与过多个大型 SaaS 项目的架构设计。
请 review 以下代码，从架构层面给出建议..."
```

## 四、常见 Web 开发场景的 Prompt 示例

### 场景 1：生成 API 路由

```
"在 Next.js App Router 中创建一个 API 路由，
路径：/api/products
功能：支持分页查询、按分类筛选、按价格排序
数据库：Prisma + MySQL
需要处理：参数验证、错误处理、日志"
```

### 场景 2：调试性能问题

```
"这个 React 组件在渲染 1000 条数据时卡顿严重。
请分析可能的原因，并给出优化方案。
注意：不能使用虚拟滚动（业务需求限制）"
```

### 场景 3：代码重构

```
"这个函数有 200 行，做了太多事情。
请按照单一职责原则重构它。
保持相同的对外接口，不要修改调用方代码。"
```

## 五、Prompt 工程的常见误区

### 误区 1：Prompt 越长越好

**不是。** 关键是**信息密度**，不是长度。100 个字的精准 Prompt 胜过 500 个字的废话。

### 误区 2：一次问完所有事情

**不要。** 分步提问。先让 AI 理解上下文，再逐步深入。

### 误区 3：不验证 AI 的输出

**永远要验证。** AI 看起来很自信，但它可能完全错了。

## 六、总结

```
Prompt 工程的核心公式：

好的输出 = 清晰的上下文 + 明确的指令 + 合理的约束 + 迭代的追问
```

**记住三点：**
1. **给 AI 当"产品经理"** — 告诉它你要什么，而不是让它猜
2. **分步迭代** — 不要一次问完，逐步深入
3. **永远验证** — AI 是工具，不是权威
