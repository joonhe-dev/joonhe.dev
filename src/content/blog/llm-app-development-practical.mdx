## 引言

2023 年初，我第一次尝试用 GPT-3.5 做一个自动回复客服系统。当时的做法很简单：把用户问题粘到 Prompt 里，拿到回复直接返回。效果嘛——客户说"你们这个 AI 客服是不是喝多了"。

一年半后的今天，我已经用 LLM 搭建了多个生产级应用，包括知识库问答系统、代码审查助手、自动化报告生成器。踩过的坑可以写一本书。

这篇文章是我对 LLM 应用开发的经验总结。我把 LLM 应用分为三个层次：

1. **Prompt 工程** — 让 LLM 准确理解你的意图
2. **RAG 架构** — 让 LLM 拥有你的私有知识
3. **Agent 模式** — 让 LLM 能自主执行任务

每个层次都有对应的代码示例和实战经验，希望能帮你少走弯路。

## 第一层：Prompt 工程

### 从"问问题"到"给指令"

大多数人对 Prompt 的认知停留在"问问题"阶段：

```typescript
// ❌ 错误做法：像问朋友一样问 LLM
const prompt = "帮我写一个 React 组件，显示用户列表"
```

但生产级的 Prompt 应该是**结构化指令**：

```typescript
// ✅ 正确做法：给 LLM 明确的角色、上下文、格式要求
const systemPrompt = `
你是一个资深 React 开发者。你的任务是生成高质量、可维护的 React 组件代码。

## 约束条件
- 使用 TypeScript + Tailwind CSS
- 包含完整的类型定义
- 遵循函数式组件 + Hooks 的最佳实践
- 包含 JSDoc 注释
- 不包含外部依赖（除了 React 和 Tailwind）

## 输出格式
返回一个包含以下内容的代码块：
1. 类型定义（interface / type）
2. 组件实现
3. 使用示例
`

const userPrompt = `
需求：显示用户列表的组件
- 从 API 获取用户数据
- 支持搜索和分页
- 每页显示 10 条
- 加载状态和空状态处理
`
```

对于需要推理的任务，让 LLM 逐步思考：

### Prompt 工程最佳实践总结

| 实践 | 说明 | 效果提升 |
|------|------|----------|
| 角色设定 | 给 LLM 一个明确的角色 | +20% 准确率 |
| 结构化指令 | 用 Markdown 格式组织 Prompt | +15% 一致性 |
| Few-shot 示例 | 提供 2-3 个示例 | +30% 质量 |
| Chain-of-Thought | 引导逐步推理 | +40% 推理准确率 |
| 结构化输出 | 指定输出格式和 schema | 减少解析错误 |
| 温度控制 | 代码生成用 0-0.2，创意用 0.7-0.9 | 按需调整 |

## 第二层：RAG 架构

### 为什么需要 RAG？

LLM 的知识截止于训练数据。如果你问它"我们公司最新的 OKR 是什么"，它不可能知道。RAG（Retrieval-Augmented Generation）解决了这个问题：**先检索相关文档，再把文档作为上下文让 LLM 回答。**

### 一个完整的 RAG 实现

```typescript
import { OpenAI } from 'openai'
import { createClient } from '@supabase/supabase-js'

// 1. 文档分块（Chunking）
function chunkDocument(text: string, chunkSize = 1000, overlap = 200): string[] {
  const chunks: string[] = []
  let start = 0

  while (start < text.length) {
    const end = start + chunkSize
    // 尽量在段落边界切分
    const chunk = text.slice(start, end)
    const lastPeriod = chunk.lastIndexOf('。')
    const actualEnd = lastPeriod > chunkSize * 0.5 ? lastPeriod + 1 : end

    chunks.push(text.slice(start, start + actualEnd))
    start = start + actualEnd - overlap
  }

  return chunks
}

// 2. 生成 Embedding 并存储
async function indexDocument(text: string, metadata: Record<string, any>) {
  const chunks = chunkDocument(text)
  const embeddings = await Promise.all(
    chunks.map(async (chunk) => {
      const response = await openai.embeddings.create({
        model: 'text-embedding-3-small',
        input: chunk,
      })
      return {
        content: chunk,
        embedding: response.data[0].embedding,
        metadata,
      }
    })
  )

  // 存储到向量数据库
  const { error } = await supabase.from('documents').insert(embeddings)
  if (error) throw error
}

// 3. 检索相关文档
async function retrieveRelevant(query: string, topK = 5) {
  // 生成查询的 embedding
  const response = await openai.embeddings.create({
    model: 'text-embedding-3-small',
    input: query,
  })
  const queryEmbedding = response.data[0].embedding

  // 向量相似度搜索
  const { data: documents } = await supabase.rpc('match_documents', {
    query_embedding: queryEmbedding,
    match_threshold: 0.7,
    match_count: topK,
  })

  return documents
}

// 4. 生成回答
async function answerWithRag(query: string) {
  const relevantDocs = await retrieveRelevant(query)

  if (relevantDocs.length === 0) {
    return "抱歉，我没有找到相关信息。"
  }

  const context = relevantDocs
    .map((doc) => `[来源: ${doc.metadata.source}]\n${doc.content}`)
    .join('\n\n---\n\n')

  const response = await openai.chat.completions.create({
    model: 'gpt-4o-mini',
    messages: [
      {
        role: 'system',
        content: `你是一个知识库问答助手。请基于以下上下文回答问题。
如果上下文中没有相关信息，请明确说"没有找到相关信息"。
引用来源时，请标注 [来源: xxx]。

上下文：
${context}`,
      },
      {
        role: 'user',
        content: query,
      },
    ],
  })

  return response.choices[0].message.content
}
```

### PostgreSQL 向量搜索配置

如果你用 Supabase（PostgreSQL + pgvector），需要先创建向量搜索函数：

```sql
-- 启用 pgvector 扩展
create extension if not exists vector;

-- 创建文档表
create table documents (
  id bigserial primary key,
  content text,
  embedding vector(1536),  -- text-embedding-3-small 的维度
  metadata jsonb,
  created_at timestamptz default now()
);

-- 创建向量索引（IVFFlat 适合百万级以下的数据）
create index on documents using ivfflat (embedding vector_cosine_ops)
  with (lists = 100);

-- 向量相似度搜索函数
create or replace function match_documents(
  query_embedding vector(1536),
  match_threshold float,
  match_count int
)
returns table (
  id bigint,
  content text,
  metadata jsonb,
  similarity float
)
language plpgsql
as $$
begin
  return query
  select
    documents.id,
    documents.content,
    documents.metadata,
    1 - (documents.embedding <=> query_embedding) as similarity
  from documents
  where 1 - (documents.embedding <=> query_embedding) > match_threshold
  order by documents.embedding <=> query_embedding
  limit match_count;
end;
$$;
```

### RAG 架构的关键决策

| 决策点 | 选项 | 推荐 |
|--------|------|------|
| Chunk 策略 | 固定大小 / 语义分块 / 递归分块 | 语义分块（按段落/标题） |
| Chunk 大小 | 256-2048 tokens | 1000 tokens（含 200 overlap） |
| Embedding 模型 | text-embedding-3-small / ada-002 / 开源模型 | text-embedding-3-small（性价比最高） |
| 向量数据库 | pgvector / Pinecone / Weaviate / Qdrant | pgvector（和 PostgreSQL 一起） |
| 检索策略 | 向量搜索 / 关键词搜索 / 混合搜索 | 混合搜索（向量 + BM25） |
| 重排序 | Cohere Rerank / 交叉编码器 | 数据量大时加 reranker |

### 检索增强策略

基础 RAG 在简单场景下够用，但生产环境需要更精细的策略：

```typescript
// 混合检索：向量搜索 + 关键词搜索
async function hybridSearch(query: string) {
  // 向量搜索
  const vectorResults = await vectorSearch(query)

  // 关键词搜索（使用 PostgreSQL 全文搜索）
  const { data: keywordResults } = await supabase
    .from('documents')
    .select('*')
    .textSearch('content', query, {
      type: 'websearch',
      config: 'chinese',
    })

  // 合并结果（加权融合）
  const merged = mergeResults(vectorResults, keywordResults, {
    vectorWeight: 0.7,
    keywordWeight: 0.3,
  })

  return merged
}

// 查询重写：将用户问题改写为更利于检索的形式
async function rewriteQuery(userQuery: string): Promise<string> {
  const response = await openai.chat.completions.create({
    model: 'gpt-4o-mini',
    messages: [
      {
        role: 'system',
        content: '将用户问题改写为更适合文档检索的形式。只返回改写后的查询，不要解释。',
      },
      {
        role: 'user',
        content: userQuery,
      },
    ],
  })

  return response.choices[0].message.content
```

## 第三层：Agent 模式

### 从"问答"到"行动"

RAG 让 LLM 能回答问题，但 Agent 让 LLM 能**执行任务**。Agent 的核心是：LLM 作为一个"大脑"，决定调用什么工具、按什么顺序调用、如何处理结果。

### 一个简单的 Agent 实现

```typescript
import { OpenAI } from 'openai'
import { z } from 'zod'

// 定义工具
const tools = {
  searchWeb: {
    description: '搜索互联网获取最新信息',
    parameters: z.object({
      query: z.string().describe('搜索关键词'),
    }),
    async execute({ query }: { query: string }) {
      return await webSearch(query)
    },
  },
  readUrl: {
    description: '读取指定 URL 的内容',
    parameters: z.object({
      url: z.string().describe('要读取的 URL'),
    }),
    async execute({ url }: { url: string }) {
      return await fetchAndExtract(url)
    },
  },
  calculate: {
    description: '执行数学计算',
    parameters: z.object({
      expression: z.string().describe('数学表达式，如 "2 + 3 * 4"'),
    }),
    async execute({ expression }: { expression: string }) {
      return String(eval(expression))
    },
  },
  sendEmail: {
    description: '发送邮件',
    parameters: z.object({
      to: z.string().describe('收件人邮箱'),
      subject: z.string().describe('邮件主题'),
      body: z.string().describe('邮件正文'),
    }),
    async execute({ to, subject, body }: { to: string; subject: string; body: string }) {
      return await emailService.send(to, subject, body)
    },
  },
}

// Agent 主循环
async function agentLoop(task: string) {
  const messages: any[] = [
    {
      role: 'system',
      content: `你是一个 AI 助手，可以调用工具来完成任务。
可用的工具：
${Object.entries(tools)
  .map(
    ([name, tool]) =>
      `- ${name}: ${tool.description}\n  参数: ${JSON.stringify(tool.parameters.shape)}`
  )
  .join('\n')}

当你需要调用工具时，返回以下格式：
{"tool": "工具名", "args": {参数对象}}

当你完成任务后，返回最终结果。`,
    },
    { role: 'user', content: task },
  ]

  const maxIterations = 10
  for (let i = 0; i < maxIterations; i++) {
    const response = await openai.chat.completions.create({
      model: 'gpt-4o',
      messages,
    })

    const content = response.choices[0].message.content

    // 尝试解析工具调用
    try {
      const toolCall = JSON.parse(content)
      if (toolCall.tool && tools[toolCall.tool]) {
        const result = await tools[toolCall.tool].execute(toolCall.args)
        messages.push({ role: 'assistant', content })
        messages.push({
          role: 'user',
          content: `工具 ${toolCall.tool} 返回：${JSON.stringify(result)}\n\n请根据结果继续。`,
        })
        continue
      }
    } catch {
      // 不是工具调用，就是最终结果
    }

    return content
  }

  return '任务执行超时'
}
```

### Multi-Agent 模式

对于复杂任务，单一 Agent 难以胜任。Multi-Agent 模式将任务分解给多个 specialized agent：

```typescript
// Multi-Agent 架构示例
class MultiAgentSystem {
  private agents = {
    researcher: new Agent({
      role: '研究员',
      tools: [webSearch, readUrl],
      model: 'gpt-4o',
    }),
    writer: new Agent({
      role: '写手',
      tools: [writeFile, searchKnowledge],
      model: 'gpt-4o',
    }),
    reviewer: new Agent({
      role: '审查员',
      tools: [codeReview, spellCheck],
      model: 'gpt-4o-mini',
    }),
    coordinator: new Agent({
      role: '协调员',
      tools: [],
      model: 'gpt-4o',
    }),
  }

  async executeTask(task: string) {
    // 协调员分解任务
    const subtasks = await this.agents.coordinator.decompose(task)

    for (const subtask of subtasks) {
      // 根据任务类型分配给对应 agent
      const agent = this.selectAgent(subtask.type)
      const result = await agent.execute(subtask.description)

      // 审查员检查结果
      const review = await this.agents.reviewer.review(result)
      if (!review.passed) {
        // 如果不通过，重做
        const revised = await agent.execute(
          `${subtask.description}\n\n审查意见：${review.feedback}`
        )
        // 更新结果
        subtask.result = revised
      } else {
        subtask.result = result
      }
    }

    // 协调员汇总结果
    return await this.agents.coordinator.synthesize(subtasks)
  }
}
```

### Cost 控制

| 策略 | 效果 | 实现方式 |
|------|------|----------|
| 模型分层 | 简单问题用便宜模型 | 用分类器判断问题复杂度 |
| 缓存 | 减少重复调用 | Redis 缓存 + 语义缓存 |
| Prompt 压缩 | 减少 token 消耗 | 去除无关上下文 |
| Batch 处理 | 非实时任务批量调用 | 攒够一批再调用 API |
| RAG 前置过滤 | 减少 LLM 调用 | 先检索，无结果才调 LLM |

## 开源工具链推荐

| 工具 | 用途 | 推荐理由 |
|------|------|----------|
| LangChain | Agent 框架 | 生态最大、文档最全 |
| LlamaIndex | RAG 框架 | 数据连接器最丰富 |
| Vercel AI SDK | 前端集成 | 流式输出、Function Calling 原生支持 |
| Dify | 可视化编排 | 适合非开发者快速搭建 |
| Flowise | 低代码 Agent | 拖拽式 Agent 构建 |
| Chroma | 向量数据库 | 轻量级、本地运行 |

## 总结

LLM 应用开发的三个层次是递进的：

1. **Prompt 工程**是基础——连 Prompt 都写不好，后面都是空中楼阁。掌握角色设定、结构化指令、Few-shot、Chain-of-Thought 这些基本功。
2. **RAG 架构**是核心——大多数生产级 LLM 应用都需要接入私有知识。Chunking 策略、Embedding 选择、检索优化是关键。
3. **Agent 模式**是进阶——让 LLM 从"回答问题"进化到"执行任务"。Function Calling 是最可靠的 Agent 实现方式。

**最后一条建议：不要为了用 AI 而用 AI。** 先想清楚你的用户真正需要什么，再选择合适的技术方案。有时候，一个简单的 if-else 规则比 GPT-4 更可靠、更便宜。
