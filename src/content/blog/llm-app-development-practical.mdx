## Introduction

In early 2023, I tried building an auto-reply customer service system with GPT-3.5. My approach was dead simple: paste the user's question into a prompt, get the reply, and return it. The result? A customer said, "Is your AI support drunk?"

A year and a half later, I've built multiple production-grade LLM applications — knowledge base Q&A systems, code review assistants, automated report generators. I've made enough mistakes to fill a book.

This article is my experience report on LLM app development. I break LLM apps into three layers:

1. **Prompt Engineering** — Making the LLM understand your intent accurately
2. **RAG Architecture** — Giving the LLM your private knowledge
3. **Agent Patterns** — Letting the LLM execute tasks autonomously

Each layer comes with code examples and real-world lessons. Hope it saves you some pain.

## Layer 1: Prompt Engineering

### From "Asking Questions" to "Giving Instructions"

Most people's understanding of prompts stops at "asking questions":

```typescript
// ❌ Wrong: asking the LLM like you'd ask a friend
const prompt = "Help me write a React component that displays a user list"
```

But production-grade prompts should be **structured instructions**:

```typescript
// ✅ Right: give the LLM a clear role, context, and format requirements
const systemPrompt = `
You are a senior React developer. Your task is to generate high-quality, maintainable React component code.

## Constraints
- Use TypeScript + Tailwind CSS
- Include complete type definitions
- Follow functional component + Hooks best practices
- Include JSDoc comments
- No external dependencies (except React and Tailwind)

## Output Format
Return a code block containing:
1. Type definitions (interface / type)
2. Component implementation
3. Usage example
`

const userPrompt = `
Requirement: A component that displays a user list
- Fetch user data from an API
- Support search and pagination
- Show 10 items per page
- Handle loading and empty states
`
```

### The Power of Few-Shot Examples

For complex tasks, giving the LLM examples beats describing rules:

```typescript
// Few-shot examples: teach the LLM your code style
```

### Chain-of-Thought

For tasks that require reasoning, guide the LLM to think step by step:

```typescript
// ❌ Ask for the result directly
const prompt = "What are the performance issues in this code?"

// ✅ Guide the LLM to analyze step by step
const prompt = `
Analyze the performance issues in the following code. Think through these steps:

1. First, determine the execution flow of this code
2. Then, identify the time complexity of each step
3. Next, pinpoint potential performance bottlenecks
4. Finally, provide optimization suggestions

Code:
${code}
`
```

In production, you need the LLM's output to be structured, not free-form text:


### Prompt Engineering Best Practices Summary

| Practice | What It Means | Improvement |
|----------|---------------|-------------|
| Role assignment | Give the LLM a clear role | +20% accuracy |
| Structured instructions | Organize prompts with Markdown | +15% consistency |
| Few-shot examples | Provide 2–3 examples | +30% quality |
| Chain-of-Thought | Guide step-by-step reasoning | +40% reasoning accuracy |
| Structured output | Specify output format and schema | Fewer parse errors |
| Temperature control | 0–0.2 for code gen, 0.7–0.9 for creative tasks | Tune as needed |

## Layer 2: RAG Architecture

### Why RAG?

LLM knowledge is frozen at training time. Ask it "What are our company's latest OKRs?" and it won't have a clue. RAG (Retrieval-Augmented Generation) fixes this: **first retrieve relevant documents, then feed those documents as context for the LLM to answer.**

### A Complete RAG Implementation

```typescript
import { OpenAI } from 'openai'
import { createClient } from '@supabase/supabase-js'

// 1. Document chunking
function chunkDocument(text: string, chunkSize = 1000, overlap = 200): string[] {
  const chunks: string[] = []
  let start = 0

  while (start < text.length) {
    const end = start + chunkSize
    // Try to split at paragraph boundaries
    const chunk = text.slice(start, end)
    const lastPeriod = chunk.lastIndexOf('.')
    const actualEnd = lastPeriod > chunkSize * 0.5 ? lastPeriod + 1 : end

    chunks.push(text.slice(start, start + actualEnd))
    start = start + actualEnd - overlap
  }

  return chunks
}

// 2. Generate embeddings and store
async function indexDocument(text: string, metadata: Record<string, any>) {
  const chunks = chunkDocument(text)
  const embeddings = await Promise.all(
    chunks.map(async (chunk) => {
      const response = await openai.embeddings.create({
        model: 'text-embedding-3-small',
        input: chunk,
      })
      return {
        content: chunk,
        embedding: response.data[0].embedding,
        metadata,
      }
    })
  )

  // Store in vector database
  const { error } = await supabase.from('documents').insert(embeddings)
  if (error) throw error
}

// 3. Retrieve relevant documents
async function retrieveRelevant(query: string, topK = 5) {
  // Generate query embedding
  const response = await openai.embeddings.create({
    model: 'text-embedding-3-small',
    input: query,
  })
  const queryEmbedding = response.data[0].embedding

  // Vector similarity search
  const { data: documents } = await supabase.rpc('match_documents', {
    query_embedding: queryEmbedding,
    match_threshold: 0.7,
    match_count: topK,
  })

  return documents
}

// 4. Generate answer
async function answerWithRag(query: string) {
  const relevantDocs = await retrieveRelevant(query)

  if (relevantDocs.length === 0) {
    return "Sorry, I couldn't find any relevant information."
  }

  const context = relevantDocs
    .map((doc) => `[Source: ${doc.metadata.source}]\n${doc.content}`)
    .join('\n\n---\n\n')

  const response = await openai.chat.completions.create({
    model: 'gpt-4o-mini',
    messages: [
      {
        role: 'system',
        content: `You are a knowledge base Q&A assistant. Answer the question based on the following context.
If the context doesn't contain relevant information, say "No relevant information found" explicitly.
When citing sources, tag them as [Source: xxx].

Context:
${context}`,
      },
      {
        role: 'user',
        content: query,
      },
    ],
  })

  return response.choices[0].message.content
}
```

### PostgreSQL Vector Search Setup

If you're using Supabase (PostgreSQL + pgvector), create the vector search function first:

```sql
-- Enable pgvector extension
create extension if not exists vector;

-- Create documents table
create table documents (
  id bigserial primary key,
  content text,
  embedding vector(1536),  -- dimension for text-embedding-3-small
  metadata jsonb,
  created_at timestamptz default now()
);

-- Create vector index (IVFFlat works well for data under a million rows)
create index on documents using ivfflat (embedding vector_cosine_ops)
  with (lists = 100);

-- Vector similarity search function
create or replace function match_documents(
  query_embedding vector(1536),
  match_threshold float,
  match_count int
)
returns table (
  id bigint,
  content text,
  metadata jsonb,
  similarity float
)
language plpgsql
as $$
begin
  return query
  select
    documents.id,
    documents.content,
    documents.metadata,
    1 - (documents.embedding <=> query_embedding) as similarity
  from documents
  where 1 - (documents.embedding <=> query_embedding) > match_threshold
  order by documents.embedding <=> query_embedding
  limit match_count;
end;
$$;
```

### Key RAG Architecture Decisions

| Decision Point | Options | Recommendation |
|----------------|---------|----------------|
| Chunk strategy | Fixed-size / Semantic / Recursive | Semantic (by paragraph/heading) |
| Chunk size | 256–2048 tokens | 1000 tokens (with 200 overlap) |
| Embedding model | text-embedding-3-small / ada-002 / open-source | text-embedding-3-small (best value) |
| Vector database | pgvector / Pinecone / Weaviate / Qdrant | pgvector (stays with PostgreSQL) |
| Retrieval strategy | Vector search / Keyword search / Hybrid | Hybrid (vector + BM25) |
| Reranking | Cohere Rerank / Cross-encoder | Add a reranker when data grows |

### Retrieval Enhancement Strategies

Basic RAG works for simple scenarios, but production needs finer strategies:

```typescript
// Hybrid retrieval: vector search + keyword search
async function hybridSearch(query: string) {
  // Vector search
  const vectorResults = await vectorSearch(query)

  // Keyword search (using PostgreSQL full-text search)
  const { data: keywordResults } = await supabase
    .from('documents')
    .select('*')
    .textSearch('content', query, {
      type: 'websearch',
      config: 'english',
    })

  // Merge results (weighted fusion)
  const merged = mergeResults(vectorResults, keywordResults, {
    vectorWeight: 0.7,
    keywordWeight: 0.3,
  })

  return merged
}

// Query rewriting: reformulate the user's question for better retrieval
async function rewriteQuery(userQuery: string): Promise<string> {
  const response = await openai.chat.completions.create({
    model: 'gpt-4o-mini',
    messages: [
      {
        role: 'system',
        content: 'Rewrite the user question into a form that works better for document retrieval. Return only the rewritten query, no explanation.',
      },
      {
        role: 'user',
        content: userQuery,
      },
    ],
  })

  return response.choices[0].message.content
}
```

## Layer 3: Agent Patterns

### From "Answering" to "Acting"

RAG lets the LLM answer questions, but Agents let the LLM **execute tasks**. The core idea: the LLM acts as a "brain," deciding which tools to call, in what order, and how to process results.

### A Simple Agent Implementation

```typescript
import { OpenAI } from 'openai'
import { z } from 'zod'

// Define tools
const tools = {
  searchWeb: {
    description: 'Search the internet for up-to-date information',
    parameters: z.object({
      query: z.string().describe('search keywords'),
    }),
    async execute({ query }: { query: string }) {
      return await webSearch(query)
    },
  },
  readUrl: {
    description: 'Read the content at a given URL',
    parameters: z.object({
      url: z.string().describe('the URL to read'),
    }),
    async execute({ url }: { url: string }) {
      return await fetchAndExtract(url)
    },
  },
  calculate: {
    description: 'Perform a math calculation',
    parameters: z.object({
      expression: z.string().describe('math expression, e.g. "2 + 3 * 4"'),
    }),
    async execute({ expression }: { expression: string }) {
      return String(eval(expression))
    },
  },
  sendEmail: {
    description: 'Send an email',
    parameters: z.object({
      to: z.string().describe('recipient email'),
      subject: z.string().describe('email subject'),
      body: z.string().describe('email body'),
    }),
    async execute({ to, subject, body }: { to: string; subject: string; body: string }) {
      return await emailService.send(to, subject, body)
    },
  },
}

// Agent main loop
async function agentLoop(task: string) {
  const messages: any[] = [
    {
      role: 'system',
      content: `You are an AI assistant that can call tools to complete tasks.
Available tools:
${Object.entries(tools)
  .map(
    ([name, tool]) =>
      `- ${name}: ${tool.description}\n  Parameters: ${JSON.stringify(tool.parameters.shape)}`
  )
  .join('\n')}

When you need to call a tool, return this format:
{"tool": "tool_name", "args": {parameter object}}

When the task is complete, return the final result.`,
    },
    { role: 'user', content: task },
  ]

  const maxIterations = 10
  for (let i = 0; i < maxIterations; i++) {
    const response = await openai.chat.completions.create({
      model: 'gpt-4o',
      messages,
    })

    const content = response.choices[0].message.content

    // Try to parse a tool call
    try {
      const toolCall = JSON.parse(content)
      if (toolCall.tool && tools[toolCall.tool]) {
        const result = await tools[toolCall.tool].execute(toolCall.args)
        messages.push({ role: 'assistant', content })
        messages.push({
          role: 'user',
          content: `Tool ${toolCall.tool} returned: ${JSON.stringify(result)}\n\nPlease continue based on the result.`,
        })
        continue
      }
    } catch {
      // Not a tool call — it's the final result
    }

    return content
  }

  return 'Task execution timed out'
}
```

### Function Calling (A More Reliable Agent Pattern)

The JSON-parsing approach above isn't reliable. OpenAI's Function Calling provides native support:

```typescript
// Using OpenAI Function Calling
const response = await openai.chat.completions.create({
  model: 'gpt-4o',
  messages: [
    { role: 'system', content: 'You are a data analysis assistant.' },
    { role: 'user', content: 'Query last month\'s sales data, analyze growth trends, and send an email to the team' },
  ],
  tools: [
    {
      type: 'function',
      function: {
        name: 'querySalesData',
        description: 'Query sales data',
        parameters: {
          type: 'object',
          properties: {
            startDate: { type: 'string', description: 'Start date YYYY-MM-DD' },
            endDate: { type: 'string', description: 'End date YYYY-MM-DD' },
          },
          required: ['startDate', 'endDate'],
        },
      },
    },
    {
      type: 'function',
      function: {
        name: 'analyzeTrend',
        description: 'Analyze data trends',
        parameters: {
          type: 'object',
          properties: {
            data: { type: 'array', items: { type: 'object' } },
          },
          required: ['data'],
        },
      },
    },
    {
      type: 'function',
      function: {
        name: 'sendEmail',
        description: 'Send an email',
        parameters: {
          type: 'object',
          properties: {
            to: { type: 'string' },
            subject: { type: 'string' },
            body: { type: 'string' },
          },
          required: ['to', 'subject', 'body'],
        },
      },
    },
  ],
  tool_choice: 'auto',
})

// Handle tool calls
const toolCalls = response.choices[0].message.tool_calls
for (const call of toolCalls) {
  const args = JSON.parse(call.function.arguments)
  switch (call.function.name) {
    case 'querySalesData':
      // Execute query
      break
    case 'analyzeTrend':
      // Execute analysis
      break
    case 'sendEmail':
      // Send email
      break
  }
}
```

### Multi-Agent Patterns

For complex tasks, a single agent might not cut it. Multi-agent patterns delegate to multiple specialized agents:

```typescript
// Multi-Agent architecture example
class MultiAgentSystem {
  private agents = {
    researcher: new Agent({
      role: 'Researcher',
      tools: [webSearch, readUrl],
      model: 'gpt-4o',
    }),
    writer: new Agent({
      role: 'Writer',
      tools: [writeFile, searchKnowledge],
      model: 'gpt-4o',
    }),
    reviewer: new Agent({
      role: 'Reviewer',
      tools: [codeReview, spellCheck],
      model: 'gpt-4o-mini',
    }),
    coordinator: new Agent({
      role: 'Coordinator',
      tools: [],
      model: 'gpt-4o',
    }),
  }

  async executeTask(task: string) {
    // Coordinator breaks down the task
    const subtasks = await this.agents.coordinator.decompose(task)

    for (const subtask of subtasks) {
      // Assign to the right agent based on task type
      const agent = this.selectAgent(subtask.type)
      const result = await agent.execute(subtask.description)

      // Reviewer checks the result
      const review = await this.agents.reviewer.review(result)
      if (!review.passed) {
        // Redo if it doesn't pass
        const revised = await agent.execute(
          `${subtask.description}\n\nReview feedback: ${review.feedback}`
        )
        // Update the result
        subtask.result = revised
      } else {
        subtask.result = result
      }
    }

    // Coordinator synthesizes the results
    return await this.agents.coordinator.synthesize(subtasks)
  }
}
```

### Cost Control

| Strategy | Effect | Implementation |
|----------|--------|----------------|
| Model tiering | Use cheaper models for simple questions | Use a classifier to judge question complexity |
| Caching | Reduce repeated calls | Redis cache + semantic cache |
| Prompt compression | Reduce token usage | Remove irrelevant context |
| Batch processing | Batch non-realtime tasks | Accumulate a batch before calling the API |
| Pre-LLM filtering via RAG | Reduce LLM calls | Retrieve first; only call LLM if no results |

## Summary

The three layers of LLM app development are progressive:

1. **Prompt Engineering** is the foundation — if you can't write good prompts, everything else is a house of cards. Master the fundamentals: role assignment, structured instructions, few-shot examples, and chain-of-thought.
2. **RAG Architecture** is the core — most production LLM apps need access to private knowledge. Chunking strategy, embedding selection, and retrieval optimization are the key levers.
3. **Agent Patterns** are the advanced tier — evolving the LLM from "answering questions" to "executing tasks." Function Calling is the most reliable way to implement agents.

**One last piece of advice: don't use AI for the sake of using AI.** Figure out what your users actually need first, then pick the right technical approach. Sometimes, a simple if-else rule is more reliable and cheaper than GPT-4.
